# Master design: replicating `train_and_test.py` in `ddrs/`

**Status:** Draft, pending user review
**Date:** 2026-05-17
**Reference:** `~/projects/ddr/scripts/train_and_test.py`, invoked as
`uv run python -m scripts.train_and_test --config-name merit_training_config.yaml`.

## Goal

Replicate DDR's `train_and_test` pipeline end-to-end in `ddrs/` such that a
**deterministic-equivalence test** holds:

> Given the same network inputs (adjacency, length, slope, attributes, q'
> streamflow forcing) and the same *fixed* MLP outputs (`n`, `q_spatial`,
> `p_spatial` per reach), ddrs's per-batch L1 loss equals DDR's per-batch L1
> loss within the f32 precision floor (~1e-6 relative).

This bar deliberately sidesteps PyTorch RNG / Adam reproducibility — those are
not part of the verification. What matters is that the forward computation
(read data → build batch → route → downsample → compute loss) produces the
same number when fed the same inputs.

## Verification protocol

Three layered checks, easiest first:

### V1 — single-batch frozen-parameter loss match (the main bar)

1. Pick one mini-batch (gauge subset, fixed time window).
2. Pin MLP outputs to a fixed `(n, q_spatial, p_spatial)` per reach (saved as
   a `.npz` from DDR or generated by a known seed).
3. In DDR: hand-feed those fixed tensors through `dmc.forward` and compute the
   training L1 loss exactly as `scripts/train.py:78-83` does.
4. In ddrs: do the same end-to-end (read same data sources, build same batch,
   route with same fixed params, downsample, L1 loss).
5. Assert `|loss_ddrs − loss_ddr| < 1e-6 * loss_ddr`.

Fixture lives at `fixtures/train_and_test/single_batch_loss/` and is
regenerated by a Python script run under DDR's `uv` venv (mirrors the existing
`scripts/export_ddr_sandbox.py` workflow).

### V2 — multi-batch forward sweep on the test period

Same protocol as V1 but over the SequentialSampler test loop (no
backpropagation). Verifies that `carry_state` between batches is wired
correctly and that the zarr output layout matches DDR's.

### V3 — full training run (not a CI test; a one-shot sanity check)

Run both pipelines for 5 epochs with fixed seeds. Compare NSE distributions
across gauges. This is NOT expected to match exactly (Adam + RandomSampler
differ between PyTorch and BURN/`rand`), but distributions should look
qualitatively similar. Used to catch silent divergence not exposed by V1/V2.

## Decomposition into 5 sub-projects

Each sub-project gets its own spec → implementation plan → subagent-driven
implementation cycle. Interfaces between them are defined here so they can be
worked sequentially (and partially in parallel where dependencies allow).

```
SP-1 static-data ────►┐
                      ├─► SP-3 dataset ─► SP-4 train ─► SP-5 test+entrypoint
SP-2 async-data ──────┘
```

### SP-1 — Static-data layer (NetCDF + CSV)

**Mirrors:** `ddr/io/readers.py::AttributesReader`,
`ddr/io/readers.py::read_gage_info`, `ddr/io/statistics.py`.

**Deliverables:**

- `src/data/store/netcdf.rs`: `AttributesStore` opening
  `merit_global_attributes_v2.nc`, returning `Array2<f32>` of shape `(F, N)`
  for a list of `Comid` and a list of attribute names. Handles the lazy-load
  pattern of `xr.open_mfdataset` (single file in MERIT's case — straight
  `netcdf::open`).
- `src/data/store/gage_csv.rs`: `GageMetaStore` with fields `STAID`,
  `DRAIN_SQKM`, `DA_VALID`, `FLOW_SCALE`, `COMID`, `COMID_DRAIN_SQKM`,
  `COMID_UNITAREA_SQKM`. The first three are required; the rest are optional
  and gate flow-scale computation.
- `src/data/statistics.rs`: per-attribute mean/std over the full CONUS column,
  matching DDR's `set_statistics`. Stored as `HashMap<String, (mean, std)>`.
  Includes the NaN/inf handling from `naninfmean` and `fill_nans`.

**Interface upward:**
```rust
pub struct StaticData {
    pub attrs: Array2<f32>,            // (F, N_conus)
    pub attr_names: Vec<String>,       // length F
    pub stats: AttrStats,              // for normalization
    pub gages: GageMetadata,           // STAID → metadata
    pub means: Array1<f32>,            // length F (for fill_nans)
    pub stds:  Array1<f32>,
}
```

**Verification:** unit test that picks 10 known COMIDs and compares
`attrs[:, idx]` to DDR's `attribute_ds[attrs_list].isel(COMID=idx)` within
f32 floor. Gage CSV test against `references/gage_info/gages_3000.csv` with
known counts.

**Concerns:**
- `netcdf` crate (v0.12) is HDF5-backed and sync. Read patterns must avoid
  the chunked-axis-mismatch slow path.
- DDR uses `xr.open_mfdataset` then `.isel(COMID=valid_indices)` —
  potentially out-of-order indices. We need to verify the Rust crate handles
  fancy indexing or we implement it ourselves with sorted reads.

### SP-2 — Async data layer (icechunk)

**Mirrors:** `ddr/io/readers.py::read_ic`,
`ddr/io/readers.py::StreamflowReader`, `ddr/io/readers.py::IcechunkUSGSReader`.

**Deliverables:**

- `src/data/store/icechunk.rs`: two structs, `StreamflowStore` and
  `UsgsObservationsStore`. Each owns an `icechunk::Repository` handle and a
  `tokio::runtime::Runtime`. Public API is sync — `block_on` at the boundary.
- Streamflow read: takes a `RhoWindow` (daily indices) + list of `Comid`,
  returns `Array2<f32>` shape `(n_hourly, n_active)`. Includes the
  daily-to-hourly `repeat(24)` trim to `n_hourly = (rho - 1) * 24` from the
  existing `RhoWindow::n_hourly`.
- Observations read: takes a `RhoWindow` + list of `Staid`, returns
  `Array2<f32>` shape `(rho_days, n_gauges)`. NaN handling preserved.

**Interface upward:**
```rust
pub struct StreamflowBatch { pub q_prime: Array2<f32> /* (T_hours, N) */ }
pub struct ObservationsBatch { pub obs: Array2<f32> /* (T_days, G) */ }
```

**Verification:** read a known window from both stores via icechunk-rs and
compare to xarray's `open_zarr` + `.isel` output. Tolerance: bit-exact (these
are read-only stores, no math).

**Concerns:**
- **Highest unknown-unknown of the project.** `icechunk` v2 Rust crate is
  much less battle-tested than the Python API. Possible gaps in chunk
  fetching, session lifetime, async ergonomics.
- icechunk repos use timestamped commits / sessions. We need read-only access
  to the `main` branch as DDR does.
- The streamflow store is daily-native (`is_hourly: false` in DDR's MERIT
  config). The repeat-and-trim is part of the read, not the dataset layer.

### SP-3 — Dataset + collate (batch construction)

**Mirrors:** `ddr/geodatazoo/merit.py::Merit`, especially
`_collate_gages` and `_build_common_tensors`.

**Deliverables:**

- `src/data/dataset.rs`: `MeritGagesDataset` holding refs to:
  `ConusAdjacencyStore`, `GagesAdjacencyStore`, `AttributesStore`,
  `StreamflowStore`, `UsgsObservationsStore`, `StaticData`, `TimeAxis`,
  `Config`.
- `dataset.len() -> usize` (= number of valid gauges after DA_VALID +
  headwater filtering).
- `dataset.collate(batch: &[Staid], window: RhoWindow) -> RoutingBatch<I>`:
  the equivalent of DDR's `_collate_gages`. Steps:
    1. Union the per-gauge subgraph COOs.
    2. Compute `active_indices` (union of edge endpoints + gauge outlets).
    3. Build `index_mapping: HashMap<conus_idx, compressed_idx>`.
    4. Build the compressed COO; sort and verify lower-triangularity (the
       invariant in `CLAUDE.md`).
    5. Build `SparseAdjacency` (the existing struct) with compressed
       `length_m` and `slope`.
    6. Build `outflow_idx` per gauge — list of compressed col-indices that
       feed each gauge outlet (used by SP-4 to scatter predictions onto
       gauges).
    7. Build `flow_scale: Vec<f32>` (length N_active) from gauge metadata.
       **This was deferred in ARCHITECTURE.md; we add it here.**
    8. Fetch attributes + normalize: `(attrs - means) / stds`, transpose to
       `(N, F)` per the MLP I/O contract.
    9. Fetch streamflow window for active COMIDs; multiply by `flow_scale`
       before handing to the engine.
    10. Fetch observations for the gauges in this batch.
- `src/data/sampler.rs`: a thin batched iterator. `RandomSampler<R: Rng>`
  for training, `SequentialSampler` for testing. **No PyTorch-bit-identical
  reproducibility goal.** Determinism via `rand` seeded from `cfg.seed` is
  enough for our verification bar.

**Interface upward:**
```rust
pub struct RoutingBatch<I: Backend> {
    pub adjacency: SparseAdjacency,                       // → MuskingumCunge::setup_inputs
    pub spatial_attributes_normalized: Tensor<I, 2>,      // (N, F) → MLP input
    pub q_prime: Array2<f32>,                             // (T_hours, N), pre-scaled
    pub observations: Array2<f32>,                        // (T_days, G)
    pub outflow_idx: Vec<Vec<usize>>,                     // gauge → compressed-cols
    pub gauge_staids: Vec<Staid>,                         // length G
    pub window: RhoWindow,
}
```

**Verification:**
- Unit test: small synthetic 3-gauge network, hand-compute compressed CSR,
  check shape and lower-triangularity.
- Integration test against DDR: pick a known seed + batch and dump DDR's
  `RoutingDataclass` to a fixture; load it in Rust and compare every field.

**Concerns:**
- Lower-triangularity preservation across subgraph union is non-obvious.
  DDR's `construct_network_matrix` calls `np.unique` which sorts; combined
  with topological order in `merit_ids` this preserves the invariant. We
  must replicate the same ordering.
- `flow_scale` computation has a fast path (`FLOW_SCALE` column in CSV) and a
  fallback. Implement both.
- `outflow_idx` semantics: for each gauge `g` with conus_idx `idx_g`,
  `outflow_idx[g]` is the list of compressed column-indices for rows where
  `coo.row == idx_g` (i.e., reaches that feed the gauge cell). DDR uses this
  to read gauge-cell predictions out of the dense `[n_active, T]` output.

### SP-4 — Training loop

**Mirrors:** `scripts/train.py::train`.

**Deliverables:**

- `src/training/loop.rs`: `train_one_epoch` and `train` functions.
- Optimizer: `burn::optim::AdamConfig` wrapping `Mlp<Autodiff<I>>`. Default
  betas/eps need a parity sanity check against PyTorch; document any
  differences inline.
- Learning-rate schedule: simple `epoch → lr` lookup from
  `cfg.experiment.learning_rate`.
- L1 loss: `daily_predictions[warmup:].sub(daily_obs[warmup:]).abs().mean()`.
- Daily downsample: port of `ddr.io.functions.downsample` — boundary trim
  `[13+tau : -11+tau]` then average over each 24-hour block.
- NaN-mask filter: skip gauges whose observations are all NaN in the window
  (matches DDR's `routing_dataclass.observations.isnull().any(dim="time")`).
- Gradient clipping: `clip_grad_norm_(parameters, max_norm=1.0)`.
- Checkpoint save: `burn::record::CompactRecorder` with `(epoch, mini_batch,
  rng_state, mlp_params, optimizer_state)`.
- Per-batch logging: NSE/RMSE/KGE + per-parameter min/median/max.

**Interface upward:**
```rust
pub fn train<I: Backend>(
    cfg: &Config,
    dataset: &MeritGagesDataset,
    mlp: Mlp<Autodiff<I>>,
    optimizer: impl Optimizer<Mlp<Autodiff<I>>, Autodiff<I>>,
) -> Mlp<Autodiff<I>>;
```

**Concerns:**
- Adam parity isn't part of the verification bar, but documenting any
  divergence in defaults matters for future work.
- BURN's `Optimizer` trait moves state through return values; the mutable
  optimizer pattern from PyTorch maps to "let optim = optim.step(...)".
- We must verify that the autograd graph is freed between mini-batches.
  BURN's tape lives on the autodiff tensors; dropping them is enough.

### SP-5 — Testing loop + entrypoint

**Mirrors:** `scripts/train_and_test.py::_test` + the `main` glue.

**Deliverables:**

- `src/training/eval.rs`: `evaluate` function that runs the SequentialSampler
  loop with `carry_state=true` on consecutive batches, accumulating
  predictions into a `(n_gauges, n_hours)` array.
- Daily downsample for the full predictions array.
- Zarr write: predictions + observations to a zarr store via `zarrs`. Layout
  matches DDR's: two arrays `predictions` and `observations`, both
  `(n_gauges, n_days)`, plus attrs.
- Metrics module: `src/training/metrics.rs` implementing NSE, RMSE, KGE
  per gauge over the post-warmup window.
- `src/bin/train_and_test.rs`: top-level binary. Reads the same
  `config/merit_training.yaml`. Phase 1: training. Phase 2: load last
  checkpoint, run evaluation, write zarr, log metrics.

**Interface upward:** CLI command:

```sh
cargo run --release --bin train_and_test -- --config config/merit_training.yaml
```

**Concerns:**
- The output zarr is read by DDR's downstream `examples/eval/evaluate.ipynb`.
  Keeping its layout faithful means notebooks work unchanged.
- Test-phase batching uses `batch_size: 15` (vs 64 training) and `rho: None`
  (full test period in a single pass per batch). Memory budget should be
  fine on CPU.

## Design calls made up front

These narrow choices avoid bike-shedding in each sub-spec:

1. **CPU-first.** `NdArray<f32>` backend for verification. GPU is a separate
   project after V1+V2 pass.
2. **`flow_scale` in scope.** Re-enters from the deferred list; required for
   gauges that don't sit at outlets.
3. **MLP, not KAN.** Existing `nn/mlp.rs` is the head; explicit fidelity
   tradeoff accepted.
4. **YAML config matches DDR's.** `config/merit_training.yaml` extends the
   existing one with training/testing fields.
5. **No `trait DataSet` / `trait Sampler`.** Concrete types per the existing
   data-layer convention; no premature unification.
6. **Sync-by-default with one tokio runtime.** Owned by the dataset; icechunk
   is the only async boundary; `block_on` at the read call site.
7. **Determinism via `rand::SeedableRng`, not PyTorch reproducibility.**
   Verification bar doesn't require it.

## Cross-cutting concerns

### Memory budget on CPU
- Training: 64 gauges × ~3000 active reaches per batch is a guess; actual
  number depends on subgraph overlap. Worst case is `union(64 subgraphs) ≈
  20-50k reaches`. Autograd tape over 89 hourly steps × 50k reaches × f32 =
  ~17 MB per intermediate × ~5 intermediates per step = ~7 GB. **This may
  not fit comfortably.** If we hit OOM, batch size reduction is the first
  knob.

### f32 accumulation
- The DDR sandbox match is at 5 reaches × 1 year of hourly steps. At real
  scale, sub-ULP order-of-summation differences in SpMV / triangular solve
  vs PyTorch's `spsolve_triangular` will produce visible drift.
- For V1 (single batch, fixed params), drift is bounded because we don't
  iterate gradient descent. Match should hold at the f32 floor.

### Adjacency lower-triangularity in compressed subgraphs
- `merit_ids` is in topological order. When we extract `active_indices =
  np.unique([rows, cols])`, sorting preserves topological order. The
  `index_mapping` is `(orig → compressed-position-in-sorted-active)`, so
  `compressed.rows >= compressed.cols` follows from `orig.rows >= orig.cols`.
- We must assert this in `RoutingBatch::new` and unit-test it. Breaking it
  silently kills the forward solver.

## Risks summarized

| Risk | Mitigation |
|---|---|
| icechunk-rs gaps (SP-2) | Spike read of one chunk before committing to the design; fall back to Python-driven export if blocked. |
| NetCDF read perf (SP-1) | Read full attribute matrix once at startup; per-batch is just `Array2::select`. |
| Adam parity (SP-4) | Document any divergence; not part of V1 verification bar. |
| Memory OOM at CONUS scale (SP-4) | Knob batch size; profile autograd tape; consider stepwise tape release. |
| Lower-triangular invariant break (SP-3) | Hard assertion + unit test; fail loudly. |
| Drift between Rust `rand` and PyTorch sampler (SP-3/SP-4) | Out of scope per verification bar. V3 (qualitative NSE distribution) catches gross drift. |

## Open assumptions

1. The five data sources exist on disk at the paths in
   `config/merit_training.yaml`. No S3 fetch needed.
2. DDR's MERIT pipeline is the only target. Lynker / hourly stores out of scope.
3. We can regenerate fixtures by running scripts under DDR's `uv` venv
   (same pattern as `export_ddr_sandbox.py`).
4. The MLP head reproduces enough of the KAN's representational capacity
   that V1 (fixed-param single-batch loss) passes; V3 NSE distributions may
   degrade. Already accepted.
